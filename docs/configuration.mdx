This page is the operational configuration guide for Milady.

## Configuration sources

Milady behavior is influenced by (in typical precedence order):

1. CLI flags (for that process invocation)
2. Environment variables
3. `~/.milady/milady.json`
4. Built-in defaults

Use this model when debugging "why did this value apply?" scenarios.

## Primary config location

Main config file:

```text
~/.milady/milady.json
```

Related state:

- `~/.milady/workspace/`
- `~/.milady/logs/`

## Recommended configuration workflow

1. Run onboarding or `milady setup`
2. Inspect current config with CLI commands
3. Make one scoped change at a time
4. Restart and validate effect

## Common CLI-based config operations

```bash
milady config get <key>
milady config set <key> <value>
milady configure
```

Use `milady --help` and `milady config --help` for command-level options.

## Provider configuration

Typical provider setup flow:

1. Add provider credentials
2. Verify provider status (`milady models`)
3. Send test prompt
4. Check logs if response fails

Provider/reference docs:

- `/model-providers`
- `/cli/models`

## Environment variable guidance

Use env vars for secrets and deployment-specific overrides.

Examples of when env vars are preferred:

- CI/CD or containerized runs
- per-environment credentials
- avoiding plaintext secrets in shared config files

For environment specifics, see `/cli/environment`.

## Security-critical settings

### API exposure

Default is local loopback. If exposing externally, require token auth.

```bash
echo "MILADY_API_TOKEN=$(openssl rand -hex 32)" >> .env
```

### Secret hygiene

- never commit real keys,
- rotate on suspected leak,
- keep examples fake/sanitized.

## Profiles and multi-environment usage

Use profiles to isolate development/testing/production-like state where needed.

This helps prevent cross-environment contamination and accidental config mutation.

## Troubleshooting configuration issues

If config appears ignored:

1. confirm effective profile/environment,
2. inspect current values via CLI,
3. restart process after changes,
4. inspect runtime logs for overridden/fallback values.
