This page explains how Milady is structured so users, operators, and contributors can reason about behavior changes safely.

## Architecture goals

Milady is designed for:

1. **Local-first operation** (state and control on your machine by default)
2. **Extensibility** (plugin-driven capabilities)
3. **Cross-surface consistency** (CLI, dashboard, desktop, mobile)
4. **Operational resilience** (recoverable startup and restart behavior)

## Layered system view

```text
Clients (CLI/TUI, Dashboard, Desktop, Mobile, Chrome Extension)
        ↓
API + Gateway (HTTP/WebSocket control surface)
        ↓
elizaOS Runtime (AgentRuntime + services + providers + actions)
        ↓
Milady Plugin + Other Plugins (knowledge, connectors, tools, etc.)
        ↓
State + Storage (~/.milady, workspace, database)
```

## Core runtime components

### 1) CLI bootstrap

`src/entry.ts` and `src/cli/run-main.ts` handle:

- profile/env initialization,
- command parsing via Commander,
- process-level error handling.

### 2) Runtime startup

`src/runtime/eliza.ts` handles:

- config load,
- onboarding decisions,
- workspace bootstrap,
- plugin resolution/registration,
- runtime initialization,
- headless/server-mode start.

For exact startup order, see `/agents/runtime-and-lifecycle`.

### 3) Plugin model

Milady runtime composes:

- first-party runtime bridge logic,
- core elizaOS plugins,
- optional/provider/connector plugins,
- local/custom plugin sources.

For plugin internals, see `/plugins/architecture` and `/plugins/development`.

## Configuration and state model

By default, Milady persists state under `~/.milady/`:

- `milady.json` (primary configuration)
- `workspace/` (agent files/context)
- `logs/` (runtime logs)

This local-state model is central to Milady's privacy and operator workflow.

## Interfaces and operating modes

Milady can run in two practical modes:

- **Interactive mode** (`milady`) for local human-in-the-loop usage.
- **Headless mode** (`milady start --headless`) for service/process-manager operation.

UI surfaces consume the same backend contract (REST/WebSocket).

## Security posture in architecture

Key principles:

- loopback-first exposure by default,
- explicit token/auth requirements for non-local exposure,
- sandbox and guardrail support for risky execution paths,
- plugin trust boundaries should be treated as security boundaries.

See `/guides/sandbox` and `/advanced/logs` for operational controls.

## Restart and lifecycle behavior

Runtime restart is a first-class operation and is used by:

- CLI restart flows,
- developer runtime swap flows,
- API-triggered lifecycle actions.

See `/agents/runtime-and-lifecycle` for precise state transitions.

## Extension points (where to build)

Use these extension surfaces depending on your goal:

- **Plugin-level capability:** `/plugins/create-a-plugin`
- **Custom action behavior:** `/guides/custom-actions`
- **Trigger/automation behavior:** `/guides/triggers`
- **Hook-based lifecycle logic:** `/guides/hooks`

## Troubleshooting architecture-level issues

If behavior is unclear:

1. Check runtime logs (`~/.milady/logs/`)
2. Verify config correctness (`/configuration`)
3. Confirm loaded plugins/runtime status (`/api-reference` + runtime endpoints)
4. Validate startup sequence assumptions against `/agents/runtime-and-lifecycle`
